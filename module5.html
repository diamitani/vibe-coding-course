Context Engineering

If prompt chaining is about asking the right question, context engineering is about giving AI the right information. The more relevant context you provide, the more accurate and aligned the output becomes.

  • Why Context Matters
  • What to Provide
  • Key Documents
  • Strategies
  • Practice

Why Context is Everything

Think of an AI tool as a brilliant new hire on their first day. They're incredibly talented, but they know nothing about your project, brand, preferences, or constraints. Without context, they'll make assumptions ‚Äî and those assumptions will be wrong.

‚ùå Without Context

"Build me a website" ‚Üí AI picks random colors, generic fonts, placeholder copy, and guesses at architecture. Result: generic, off-brand, requires heavy revision.

‚úÖ With Context

"Build me a website" + brand guide + design system + PRD + competitor examples ‚Üí AI matches your brand, follows your patterns, uses correct copy. Result: production-ready output.

üìê The Context Equation

Better Context = Better Output. This is the single most impactful thing you can learn in this course. Spending 30 minutes preparing context will save you hours of iteration.

Types of Context You Can Provide

üìÅ Files & Data

  • Existing code ‚Äî Upload your current codebase so AI understands your stack, patterns, and conventions
  • Databases/schemas ‚Äî Table definitions, relations, sample data
  • Content ‚Äî Real copy, product descriptions, user data (anonymized)
  • Assets ‚Äî Logos, images, icons for visual reference

üìù Documents

  • PRDs (Product Requirements Documents) ‚Äî What to build, why, and acceptance criteria
  • Product Specs ‚Äî Feature details, user stories, edge cases
  • Design Systems ‚Äî Colors, typography, spacing, component patterns
  • Brand Guidelines ‚Äî Voice, tone, visual identity, do's and don'ts

üñºÔ∏è Visual References

  • Screenshots ‚Äî of existing UI, competitor sites, inspiration
  • Wireframes/Mockups ‚Äî rough sketches or Figma exports
  • Style references ‚Äî "I want it to look like this site" (with link or screenshot)

Creating Key Documents

The PRD (Product Requirements Document)

A PRD tells the AI what to build and why. Here's a template:

PRD Template
# Product: [Name]
# Goal: [What this product/feature does]
# Target User: [Who is this for]

## Features
1. [Feature name] ‚Äî [description, priority]
2. [Feature name] ‚Äî [description, priority]
3. [Feature name] ‚Äî [description, priority]

## User Stories
- As a [user], I want to [action] so that [benefit]
- As a [user], I want to [action] so that [benefit]

## Acceptance Criteria
- [ ] [Criteria 1]
- [ ] [Criteria 2]

## Out of Scope
- [What we're NOT building]

The Design System Document

A design system tells the AI how things should look:

Design System Template
# Design System

## Colors
Primary: #8B5CF6
Secondary: #06B6D4
Background: #0A0A0F
Text: #F0F0F5
Muted: #9494A8

## Typography
Font: Inter
Headings: 700-900 weight
Body: 400 weight, 16px, 1.7 line-height

## Spacing
Base unit: 8px
Card padding: 24px
Section gap: 64px

## Components
Border radius: 12px (cards), 9999px (buttons)
Shadows: soft, dark, subtle glow on hover
Transitions: 300ms ease-out

Brand Guidelines Document

Brand guidelines tell the AI who you are:

Brand Guidelines Template
# Brand: [Name]

## Voice & Tone
- Professional but approachable
- Confident, not arrogant
- Clear and concise, no jargon

## Visual Identity
- Modern, clean, minimal
- Dark mode preferred
- Purple/cyan accent palette

## Do's
- Use action verbs in CTAs
- Include social proof

## Don'ts
- No stock photos of handshakes
- No walls of text
- Never use Comic Sans

Context Management Strategies

1. The Context Stack

Layer your context from most general to most specific:

  1. Brand guidelines (rarely changes)
  2. Design system (changes occasionally)
  3. PRD (changes per project)
  4. Specific prompt (changes per interaction)

2. Context Window Management

AI models have limited context windows. When working with large codebases:

  • Be selective ‚Äî Don't dump everything. Provide only the relevant files.
  • Summarize ‚Äî For large documents, provide a summary + the specific sections that matter.
  • Use references ‚Äî Instead of including full files, reference them: "Follow the same pattern as the UserCard component."

3. The Example Technique

One of the most powerful context strategies: show, don't just tell.

Example-Based Prompt
"Build a feature card component. Here's an example of our existing card style:

[paste screenshot or code of existing card]

Match this exact style: same border radius, same padding, same hover animation. The new card should have an icon, title, description, and a tag."
